# v0.7.0 — Render-QA Inspector（渲染质检员）

Date: 2026-06-13 · 基线：v0.6.5（含未发布的 v0.6.6 `--preview-outline`/`--confirm-outline` 检查点）

## 为什么改（不是改了什么）

渲染是红海：frontend-slides、guizang-ppt-skill 这类模板库都在卷「AI 把 PPT 渲染得多好看」，而且卷得很好。没人占的位置是**渲染完了谁来盯**——渲染产物有没有占位残留、动画有没有整体熄火、布局契约有没有被下游发明掉。Humanize 从 v0.6.4 起就有这条 QA 循环腿（`--qa-from`，3 轮封顶，fix prompt 交回下游），v0.7.0 做的事是把这个身份**做穿**：从「简报编排器，顺便盯 QA」推进成「渲染质检员，编排是它的上半场」。

五条 invariant 不动：brief-only、下游 100% 原生、QA 3 轮封顶、speaker notes 归下游、brief 是下游唯一读物。

## 四件事

### 1. 英文渲染器明文降级（诚实，不是修英文）

`registry/renderer_registry.json` 新增 `support_level` 字段：

- `guizang` → `full`：brief 出口 + QA 循环都在真实渲染产物上验证过；
- `frontend-slides` / `beautiful-html-templates` → `brief-only`：brief 出口可用，`--qa-from` 这条腿没在它们的真实产物上跑过，**不承诺**。

README 中英文各加一节照实写清。这是边界声明，不是待办——等第一单真实英文渲染产物走完 QA 循环再升级。多媒体同理：`media.video` / `video_slots.json` 决定字段维持，管线归下游，本仓库不修。

### 2. 演讲 QA 大纲（质检员的第一个自有可截图产物）

`scripts/preview_outline_html.py`（独立小脚本，零侵入 `humanize_ppt_v2.py`，纯标准库）：输入 `slide_plan.json`，输出一页零依赖单文件 HTML——观众状态转移图，每页一行「页号 → 观众进入状态 → 本页意图 → 离开状态」，顶部一条状态弧摘要。与 v0.6.6 的 `--preview-outline`（markdown 检查点）是同一道关的两种形态：markdown 给 Agent 读，HTML 给人看、给截图。

真实样例（非摆拍）：`examples/04-preview-outline-ai-tool-update/`——`examples/01-ai-tool-update/source.md` 跑 brief 模式得到的 `slide_plan.json` + 生成的 HTML + 截图，截图已嵌入 README。

### 3. QA 失败模式目录分层

`references/qa-failure-modes.md` 重排为两层：渲染器无关的**失败类别层**（占位残留 / 动效退化 / 布局契约违约 / 背景层缺失 / AI 草稿痕迹，全部映射到脚本里真实存在的规则），加 guizang 专属模式目录（症状 / 检测规则名 / fix prompt 方向）。明文班规：**只列代码里真实存在的规则**；文本溢出、对比度、视口截断等真实但未实现静态检测的类别，如实标注「不在扫描范围内」，不冒充模式。英文渲染器小节维持 reserved 并写明原因（QA 腿未验证）。

### 4. README / SKILL.md 重定位

- hero 钩子：「所有人都在教 AI 把 PPT 渲染得好看，没人盯着它渲染完有多烂。」
- 新增 showcase 空位节「QA 翻车现场：before/after」——等第一单真实 QA 抓到的翻车案例，宁空不摆拍。
- SKILL.md description 补质检员定位 + 触发词（「帮我盯一下渲染出来的 PPT 有没有翻车」「PPT 渲染质检」）+ 负触发（只要漂亮模板 / 直接渲染 → 指路 guizang / frontend-slides 类）。

## 边界（v0.7.0 仍然不做）

- 不渲染、不 post-process 下游 HTML；
- 不给英文渲染器发明 QA 规则；
- 不在 showcase 里放合成案例；
- 不接管多媒体管线。
