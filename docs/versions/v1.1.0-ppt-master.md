# v1.1.0 — PPT Master native PPTX compatibility

Humanize can now hand its AST story to `hugohe3/ppt-master` and check the resulting native `.pptx` without taking over PPT Master's visual system.

## Version boundary

English renderer support already reached `full` in v1.0. v1.1 does not introduce a second or reduced English path; it adds PPT Master as the native editable `.pptx` renderer for both Chinese and English. The paired five-slide showcases keep page structure and production gates constant while intentionally diverging in visual language: Memphis Pop for Chinese and Risograph Zine for English.

## Added

- `--renderer ppt-master`
- `ppt-master-production-prompt.md` downstream entrypoint
- `ppt-master-source.md` semantic page/notes/media contract
- disposable `outputs/ppt-master-handoff/` copies for PPT Master's main-pipeline `import-sources --move` rule
- raw `.pptx` template routing through `--ppt-master-template` → native `template-fill-pptx`
- PPT Master canvas/project/style/native-object/transition/animation/visual-review flags
- Python 3.10+ preflight and `--ppt-master-python` override, so macOS system Python 3.9 is not emitted into copy-paste commands
- renderer-native style-gate delegation to PPT Master's mandatory three-stage Confirm UI
- `scripts/pptx_qa.py` OOXML presentation checkup with 11 renderer-specific failure modes

## Preserved boundaries

- Humanize does not import PPT Master templates or author `design_spec.md` / `spec_lock.md`.
- PPT Master's Strategist confirmation is still mandatory.
- PPT Master remains responsible for sequential SVG authoring, its quality gates, notes packaging, and native export.
- Humanize never patches exported OOXML; `fix_prompt.md` sends corrections back to PPT Master's owning source.
- A raw `.pptx` template never enters the SVG template route.
