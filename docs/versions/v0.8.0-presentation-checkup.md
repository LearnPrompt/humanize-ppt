# v0.8.0 — 演讲体检（Presentation Checkup）

Date: 2026-06-13 · 基线：v0.7.0（渲染质检员定位）

## 为什么改（不是改了什么）

v0.8.0 是一次定位级返工，全部来自 v0.7.0 的人工验收。验收给了四条判断，每条对应一项整改：

### 判断一：QA 这个词要换概念

原话：「QA 是让用户准确地表达他们的问题：PPT 页跟原来大纲页有什么差异，然后才能产出能用来演讲的 HTML PPT——这是有演讲模式之后最能说明的一句定语」「大家会被 HTML 的多样式吸引沉迷，结果做出来一页里面什么内容都没有——那只是适合拿来看的 HTML，我们要做的是拿来演讲的」。

整改：文档层面「QA 循环」统一更名为「演讲体检」，首次出现处标注（即 QA 循环，CLI 仍是 `--qa-from`）。CLI flag、函数名、文件名（`qa_report.md` 等）全部不动，保持兼容。给它一句定语，hero 和安装节都出现：**演讲体检对的不是美观，是大纲：逐页核对渲染结果和大纲页的差异，把「只能看、不能讲」的页揪出来，直到每一页都拿得出口去讲。**

### 判断二：反向指路的观点不对，应该是热插拔

原话：「本质上我们广义兼容所有可以输出 HTML PPT 的 skill，只是我们跑通了这两三个、输出比较稳定，所以推荐。」

整改：README 路线节改写为热插拔双推荐。brief 是普通 markdown+JSON，谁都能读，所以广义兼容任何能产出 HTML PPT 的下游 skill；真实跑通且输出稳定的两条作为推荐：中文 → guizang-ppt-skill，英文 → frontend-slides / beautiful-html-templates；其他下游可以接，跑通欢迎反馈。SKILL.md description 里「Do NOT use … route those to …」的否定式指路改成选择式：只想要一页漂亮模板、不需要大纲和演讲体检时，直接用渲染类 skill 即可。

### 判断三：示例 prompt 对用户太难

原话：安装使用的示例 prompt 全英文术语过多（「brief 看起来 OK。请按 prompt 调 guizang-ppt-skill 原生渲染中文 deck (Style A)。渲染完用 Humanize PPT --qa-from 跑 QA…」这种）。

整改：README 首屏使用区重写成三步白话，每步一句可直接复制的中文：「把这份材料做成中文演讲 PPT，先用 humanize-ppt 出大纲和每页意图」「按这份大纲，用 guizang-ppt-skill 渲染成 deck」「渲染完跑一遍演讲体检，告诉我哪几页只能看不能讲，给出修复指令」。英文路线同样三句。`--qa-from`、Style A、fix_prompt 全部挪进「进阶用法」折叠节，首屏使用区零 CLI 参数、零英文术语（产品名除外）。

### 判断四：失败的页要用完整句子向读者说清楚

原话：「不要用——号来表达概念」「一个页面里只有几个字，甚至没有表达完整的意思，或者没有实现观众的转移，它是一个不应该存在的页。」

整改：`references/qa-failure-modes.md` 重写为中文白话目录，开头用完整句子定义失败的页；每条模式补一行「观众视角会看到什么」（如：占位残留就是观众看到 lorem/TODO 字样）；全文不再用「——」挂概念。

## 英文链路验真（v0.7.0 的边界声明兑现了一半）

v0.7.0 把英文路径明文标为 `brief-only`，并说「等第一单真实英文渲染产物走完 `--qa-from` 再升级」。v0.8.0 兑现：

- 对 `docs/showcase/hermes-agent-mastery/en/ppt/index.html`（Neo-Grid Bold，11 页，beautiful-html-templates 原生渲染）真实执行了演讲体检。
- 第 1 轮静态扫描 pass（0 fail / 0 warn；同日负向对照：注入 lorem/TODO 的副本扫出 2 条 fail 并产出指向 beautiful-html-templates 的 fix_prompt，证明 pass 不是空转）。
- 截图逐页复核发现页码徽章遮挡 9 页正文（观众看到「uires confirmation.」这样的断句），属于典型的「只能看、不能讲」。修复保持 Neo-Grid 视觉体系（只给被遮挡文字留 96px 空位），第 2 轮静态扫描 + 截图复检通过。
- 逐轮记录：`docs/showcase/hermes-agent-mastery/en/qa/presentation-checkup-2026-06-13.md`。
- 据实升级：`beautiful-html-templates` → `brief+qa-verified`（不升 `full`，因为它的专属失败模式规则仍为 0 条）；`frontend-slides` 维持 `brief-only` 并写明原因（没有它的真实渲染产物过过体检）。

配套代码改动（不破坏任何 CLI/函数/文件名）：

- `placeholder-residue` 升级为渲染器无关规则（scope `any`），检测扩展到 lorem ipsum / TODO / TBD，让体检对英文渲染器不再空转；
- `fix_prompt.md` 的 renderer 字段不再硬编码 guizang，如实写实际渲染器；
- 新增回归测试：英文渲染器的渲染器无关检查（`test_qa_loop_runs_renderer_agnostic_checks_for_english_renderer`）和已体检英文 deck 的常绿锁（`test_english_showcase_deck_passes_presentation_checkup`）。

## 边界（v0.8.0 仍然不做）

- 不渲染、不 post-process 下游 HTML，修复永远走 fix prompt 交回下游；
- 体检 3 轮封顶，超出 `needs-human`；
- 截图逐页复核是体检方法论的一半，但今天没有自动化：文字溢出、遮挡、对比度这类需要真实浏览器渲染的失败类，静态扫描如实标注「测不出」，不冒充模式；
- 多媒体管线归下游：`media.video` / `video_slots.json` 决定字段维持，渲染、占位、嵌入不归本仓库。
